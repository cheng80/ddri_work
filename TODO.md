# DDRI Top-Level TODO

작성일: 2026-03-14  
목적: `works/`와 `cheng80/` 전반에 흩어진 문서·노트북·산출물을 줄이고, 앱 제작 직전 기준의 핵심 근거 자료만 남기는 정리 작업을 별도 트랙으로 관리한다.

## 배경

현재 프로젝트는 분석 자체보다 문서와 파일 구조를 파악하는 데 시간이 많이 든다.

- `works/` 안에 단계별 문서, 중간 산출물, 요약 문서가 누적되어 있다
- `cheng80/` 안에도 API 검증, 군집 실험, 오류 분석 자료가 분산되어 있다
- 실제 최종 목표는 앱 제작 전 `핵심 노트북 2~3개` 안에서 분석 근거와 서비스 연결 논리를 일괄 확인할 수 있는 상태다

즉, 이 TODO의 목적은 새 분석을 늘리는 것이 아니라 `핵심 자료 압축`이다.

## 최종 목표

앱 제작 직전 기준으로 아래 수준까지 정리한다.

1. 핵심 노트북은 최대 `2~3개`로 압축
2. 최종 판단에 직접 쓰지 않는 중간 요약 문서와 중복 문서는 별도 보관 또는 제거
3. 핵심 CSV/이미지/룩업 테이블만 남기고 나머지 산출물은 재생성 가능 여부 기준으로 정리
4. 새 작업자가 `README -> 핵심 노트북 -> 핵심 산출물` 순서만 따라가도 전체 흐름을 이해할 수 있게 만든다

## 압축 대상 핵심 노트북 후보

현재 기준 후보는 아래와 같다.

1. `works/05_prediction_long` 대표 대여소 예측 및 군집 실험 핵심 노트북 1개
2. `works/06_prediction_long_full` 전체 161개 baseline 및 오류 분석 핵심 노트북 1개
3. `cheng80/01_ddri_api_verification.ipynb` 또는 서비스 연결용 API/lookup 정리 노트북 1개

최종적으로는 이 3개 안팎에서 아래가 모두 보여야 한다.

- 예측 데이터 기준
- 대표 15개 및 전체 161개 모델 결과
- 군집별 커스텀 실험 결론
- 오류 스테이션 해석
- 실시간 API 매핑 및 예외 규칙
- 앱 제작 직전 필요한 입력/출력 근거

## 실행 순서

1. 정본 문서와 중복 문서를 구분한다
2. 각 폴더별 `보존 / 축소 / 보관 / 삭제 후보` 목록을 만든다
3. 최종 핵심 노트북 2~3개 후보를 확정한다
4. 산출물 경로를 그 핵심 노트북 기준으로 다시 묶는다
5. 상위 README와 overview 문서를 새 구조 기준으로 다시 정리한다

## 원칙

- 노트북은 설명 가능한 정본만 남긴다
- `.py`는 재생성 편의용일 때만 유지한다
- CSV는 `output/data/`처럼 목적별 산출물 경로에만 둔다
- 중복 요약 문서는 줄이고, 정본 문서에서 링크로 연결한다
- 삭제 전에는 반드시 `재생성 가능 여부`를 확인한다

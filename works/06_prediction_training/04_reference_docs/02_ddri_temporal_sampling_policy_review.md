# DDRI 시계열 중복 축소 샘플링 검토

작성일: 2026-03-17

## 1. 질문

시계열 데이터에서 비슷한 일, 주, 월이 반복될 때 중복 학습을 줄이기 위해:

- 일단위 유사 패턴 제거
- 주단위 유사 패턴 제거
- 월단위 유사 패턴 제거

를 순차적으로 적용해, 원본은 유지하되 별도 학습 데이터에서 행을 실제로 줄이는 방식이 가능한가를 검토한다.

## 2. 결론

가능하다.

다만 이 방식은 일반적인 정본 구축 방식이라기보다, 학습용 축소 데이터셋을 만드는 `샘플 선택` 또는 `대표 구간 추출`에 가깝다.

따라서 적용 위치는:

- 정본 `canonical_data`: 적용하지 않음
- 학습용 파생 데이터: 필요 시 적용 가능

## 3. 관련 개념

가까운 실무 개념은 아래와 같다.

- instance selection
- prototype selection
- representative segment extraction
- redundancy reduction
- temporal clustering based sampling

## 4. 기대 장점

- 반복 구간이 과도하게 많은 경우 학습 시간을 줄일 수 있다.
- 매우 비슷한 패턴의 과대표집을 줄여 특정 패턴에 과도하게 맞는 현상을 완화할 수 있다.
- 실험용 축소셋을 빠르게 만들어 모델 후보를 먼저 탐색할 수 있다.
- 대표 패턴 중심으로 데이터 구조를 요약하기 쉬워진다.

## 5. 주요 단점

- 계절성, 휴일 효과, 기상 효과, 이상치 전후 맥락까지 함께 삭제될 수 있다.
- “비슷해 보이지만 중요한 차이”를 잃기 쉽다.
- 예측에서 필요한 희귀 구간의 빈도가 더 낮아질 수 있다.
- 일, 주, 월을 연속 제거하면 데이터 분산이 인위적으로 줄어 성능이 과대평가될 수 있다.
- 시계열의 실제 발생 빈도를 왜곡해 운영 환경과 다른 학습 분포를 만들 수 있다.

## 6. 특히 주의할 점

아래 구간은 유사도만 보고 제거하면 손실 위험이 크다.

- 강수 또는 강풍 구간
- 공휴일, 연휴, 명절 전후
- 급격한 수요 변화가 있었던 시점
- 신규 대여소 초기 운영 구간
- 장기 추세 전환이 있었던 기간

즉, 단순 거리 기준만으로 기계적으로 삭제하는 방식은 위험하다.

## 7. 삭제형 샘플링을 쓴다면 권장 원칙

원칙은 아래가 안전하다.

1. 정본은 유지한다.
2. 삭제는 학습용 파생 데이터에서만 수행한다.
3. 검증 `2024`, 테스트 `2025`는 삭제하지 않는다.
4. 삭제 대상은 `2023 train` 내부에 한정한다.
5. 날씨, 휴일, 이벤트 구간은 보호 규칙을 둔다.
6. 삭제율은 작게 시작한다.

## 8. 권장하지 않는 방식

아래 방식은 현재 단계에서 권장하지 않는다.

- 일 제거 후 다시 주 제거 후 다시 월 제거를 모두 연쇄 적용
- 전 기간에 동일 규칙으로 대량 삭제
- `bike_change` 절대값만 보고 단순 0 근처 구간을 일괄 삭제
- 검증셋과 테스트셋까지 같은 규칙으로 축소

이유는 정보 손실과 분포 왜곡이 너무 크기 때문이다.

## 9. 추천 대안

실무적으로는 아래 순서가 더 안전하다.

### A. 무샘플링

가장 기본 기준선이다.

- 정본 그대로 학습
- 성능과 학습시간을 기준으로 이후 축소 필요성을 판단

### B. 가중치 방식

유사 구간을 삭제하지 않고 가중치만 낮춘다.

- 정보는 유지
- 과대표집 영향만 완화
- 분포 왜곡이 삭제형보다 작음

현재 상황에서 가장 안전한 1차 대안이다.

현재 프로젝트 기준 가중치 2안은 아래처럼 정의할 수 있다.

- 대상: `2023 train`
- 단위: 같은 `station_id`, 같은 `weekday`
- 프로파일: 일단위 24시간 `bike_change_deseasonalized` 또는 `bike_change_raw`
- 처리:
  - 유사한 일들을 군집화
  - 같은 군집에 많이 몰린 날짜일수록 낮은 가중치 부여
  - 행은 삭제하지 않음
- 기본 가중치 예시:
  - `1 / sqrt(cluster_size)`
  - 마지막에 전체 평균이 1이 되도록 정규화

### C. 일단위 대표일 추출

실제로 행을 줄여야 한다면 이 방식을 먼저 권장한다.

- 스테이션별 일단위 24시간 프로파일 생성
- 같은 요일끼리만 비교
- 거리 기준으로 유사한 일들을 군집화
- 각 군집에서 대표일 1개만 남김
- 나머지 일은 학습셋에서 제거

이 방식은 주, 월까지 연쇄 삭제하는 것보다 통제가 쉽다.

## 10. 행을 실제로 줄이는 방식으로 갈 때 추천안

현재 기준 추천안은 아래와 같다.

### 추천안 1. 일단위만 축소

가장 권장한다.

- 대상: `2023 train`만
- 단위: `station_id + date`
- 비교 기준: 같은 `station_id`, 같은 `weekday`
- 프로파일: 24시간 `bike_change_raw` 또는 `bike_change_deseasonalized`
- 거리: 표준화 후 Euclidean distance 또는 cosine distance
- 처리: 작은 거리 군집에서 대표일 1개만 남김

장점:

- 해석이 쉽다.
- 삭제 범위를 통제하기 쉽다.
- 주/월 구조를 직접 깨뜨리지 않는다.

### 추천안 2. 일단위 축소 + 보호 규칙

실무 적용시 더 적절하다.

보호 대상:

- 강수일
- 공휴일
- 월말/월초
- 이상치 상위 구간

이 구간은 삭제 후보에서 제외한다.

### 추천안 3. 삭제율 상한 적용

아래 상한을 두는 것이 안전하다.

- 초기 실험: train의 10% 이내 축소
- 확대 실험: 최대 20% 내

그 이상은 왜곡 가능성이 커진다.

## 11. 비추천안

현재 단계에서 아래는 비추천이다.

- 일단위 제거 후 주단위 제거 후 월단위 제거를 모두 수행
- train의 30% 이상 대량 축소
- 스테이션 구분 없이 전역 유사도 비교

## 12. 최종 권고

현재 프로젝트에서는 아래 순서를 권장한다.

1. `무샘플링` 기준선 확보
2. `가중치 방식` 비교
3. 그래도 행 축소가 필요하면 `일단위 대표일 추출`만 제한적으로 적용
4. 주단위, 월단위 연쇄 제거는 보류

즉, 행을 실제로 줄이는 방식으로 간다면:

- 정본은 그대로 두고
- `2023 train`에만
- 같은 스테이션, 같은 요일 기준
- 일단위 대표일 추출을 소규모로 적용

이 방식이 가장 방어적이고 실험 통제가 쉽다.

## 13. 샘플링 1안 구현 메모

샘플링 1안에서 대표일 수가 목표보다 적어지는 빈 클러스터 현상은 피해야 한다.

현재 구현 기준:

- `MiniBatchKMeans` 대신 `KMeans` 사용
- 군집 중심에 가장 가까운 날짜를 대표일로 선택
- 예외적으로 선택 수가 부족하면 거리 기준으로 대표일을 추가 보충
- 그룹별 유지 개수는 `ceil(len(group) * retain_ratio)`로 계산

즉, 목표는 "대략 70% 유지"가 아니라 "각 그룹에서 최소 70% 이상 유지"에 가깝다.

## 14. 샘플링 1안 현재 행 수

기본 설정:

- 대상 타깃: `bike_change_deseasonalized`
- `retain_ratio = 0.7`
- `min_days_per_group = 8`

현재 계산 결과는 아래와 같다.

| dataset | 원본 train 2023 행 수 | 샘플링 1안 행 수 | 감소 행 수 | 실제 유지율 |
|---|---:|---:|---:|---:|
| rep15 | 131,400 | 93,600 | 37,800 | 0.712329 |
| full161 | 1,410,360 | 1,004,640 | 405,720 | 0.712329 |

유지율이 `0.7`보다 약간 큰 이유는 그룹별 대표일 수를 `ceil`로 올림 계산하기 때문이다.

## 15. 샘플링 후 반드시 다시 해야 하는 일

샘플링은 행 수를 줄이는 전처리일 뿐이다.

따라서 샘플링이 끝나면 아래 사이클을 다시 전부 수행해야 한다.

1. 샘플링된 `2023 train`으로 모델 학습
2. 원래 `2024 valid`로 후보 모델 비교
3. 우세 모델 선택
4. 샘플링된 `2023 train` + 원래 `2024 valid`로 재학습
5. 원래 `2025 test`로 최종 평가

즉, 샘플링 후 성능을 보지 않으면 샘플링의 실익을 판단할 수 없다.

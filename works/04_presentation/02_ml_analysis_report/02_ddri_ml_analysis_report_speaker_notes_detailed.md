# 강남구 따릉이 머신러닝 분석 보고서 스피치 노트

작성일: 2026-03-15

## 챕터 1. 프로젝트 개요

### 실제 스피치

“이번 보고서는 강남구 따릉이 대여소별 자전거 수요 예측 과정을 하나의 흐름으로 정리한 문서입니다.  
핵심은 군집화로 대여소의 공간 역할을 먼저 해석하고, 그다음 대표 15개 스테이션에서 피처 반응을 탐색한 뒤, 최종적으로 강남구 전체 161개 공통 스테이션 기준 운영 모델을 고르는 것입니다.  
즉 이 문서는 단순히 모델 점수만 정리한 게 아니라, 왜 이런 피처를 선택했고 왜 이 모델을 운영 기준선으로 채택했는지를 설명하는 분석 정본입니다.”

### 유의점

- 첫머리에서 `군집화 -> 대표 15개 -> 161개 운영안` 흐름을 먼저 고정한다.
- 보고서 목적은 `최종 운영 모델 선정 근거 정리`라는 점을 강조한다.

## 챕터 2. 문제 정의

### 실제 스피치

“우리가 풀고자 한 문제는 강남구 따릉이 스테이션별 자전거 수요를 예측하는 것입니다.  
여기서 예측 단위는 `station-hour`, 즉 대여소-시간 단위입니다.  
단순히 어느 대여소가 인기 있는지를 보는 것이 아니라, 특정 시간대에 어느 정도 대여가 발생하는지를 예측해야 이후 잔여 자전거 수, 부족 위험, 재배치 우선순위까지 연결할 수 있습니다.  
그래서 이 문제는 단순 분석이 아니라 서비스 운영과 직접 연결되는 예측 문제로 봤습니다.”

### 유의점

- `station-hour`를 꼭 쉬운 말로 풀어 설명한다.
- 이 챕터에서는 서비스 연결 가능성을 짧게만 언급하고, 상세 설명은 뒤로 넘긴다.

## 챕터 3. 데이터 설명

### 실제 스피치

“데이터는 팀이 확보한 강남구 따릉이 이용 데이터와 날씨·정적 입지 데이터입니다.  
학습 구간은 2023년, 검증 구간은 2024년, 최종 평가는 2025년으로 나눴습니다.  
대표 15개 실험은 군집별 대표 스테이션만 추린 탐색용 데이터셋이고, 최종 운영 모델은 2023년부터 2025년까지 공통 운영된 161개 스테이션 전체를 기준으로 평가했습니다.  
즉 15개는 설명과 탐색, 161개는 실제 운영 모델 선택이라는 역할 차이가 있습니다.”

### 유의점

- `15개와 161개의 역할 분리`를 여기서 한 번 명확히 말한다.
- 데이터 출처 세부 파일명보다 `탐색용 / 운영용` 구분이 더 중요하다.

## 챕터 4. 전처리 및 피처 엔지니어링

### 실제 스피치

“전처리에서는 시간 피처, 날씨 피처, 정적 입지 피처, 그리고 상호작용 피처를 단계적으로 만들었습니다.  
예를 들어 `lag`는 과거 시점 값, `rolling`은 최근 구간의 이동 평균이나 이동 표준편차입니다.  
정적 입지 피처에는 지하철 거리, 버스정류장 수, 대여소 표고 같은 값이 들어갑니다.  
또 `weather_full`처럼 비와 강수량을 출퇴근, 점심, 야간 시간대와 결합한 상호작용 피처도 실험했습니다.  
이 챕터의 핵심은, 단순히 원본 컬럼을 그대로 넣은 것이 아니라 운영 상황을 더 잘 설명하는 피처 조합을 단계적으로 만들었다는 점입니다.”

### 유의점

- `lag`, `rolling`, `weather_full`은 쉬운 말로 바로 풀어준다.
- 너무 많은 피처명을 나열하지 말고 대표 예시만 든다.

## 챕터 5. 군집화 분석

### 실제 스피치

“군집화는 최종 모델을 바로 고르는 단계가 아니라, 대여소의 공간 역할을 먼저 해석하는 단계였습니다.  
강남구 대여소를 5개 군집으로 나눠서, 업무 집중형인지, 주거 도착형인지, 생활·상권 혼합형인지 같은 성격을 먼저 정리했습니다.  
이후 각 군집에서 대표 스테이션 3개씩, 총 15개를 뽑아 예측 실험으로 연결했습니다.  
즉 군집화 결과는 예측 실험을 위한 상위 기준이자, 나중에 왜 어떤 피처가 중요했는지를 설명하는 해석 축입니다.”

### 유의점

- `군집화가 예측의 전처리이자 상위 기준`이라는 점을 강조한다.
- PCA 산점도, 히트맵, 지도는 `분리도 확인`, `특성 요약`, `공간 배치 확인`으로 역할을 나눠 설명한다.

## 챕터 6. 예측 모델 설계

### 실제 스피치

“예측 모델 설계에서는 먼저 비교 모델과 학습 목표 함수를 정리했습니다.  
모델은 `LightGBM`과 `CatBoost`를 비교했고, 학습 목표 함수는 `RMSE objective`와 `Poisson objective`를 비교했습니다.  
같은 모델이라도 어떤 목표 함수로 학습하느냐에 따라 결과가 달라질 수 있기 때문입니다.  
또 이 단계에서 `subset`, 즉 축소 피처 조합 실험과 `routing`, 즉 군집 분기 적용 실험, 그리고 `weather interaction` 실험 같은 비교 축도 함께 잡았습니다.”

### 유의점

- `모델`과 `학습 목표 함수`를 구분해서 설명한다.
- `subset`, `routing`은 쉬운 말로 바로 병기한다.

## 챕터 7. 대표 15개 스테이션 실험

### 실제 스피치

“대표 15개 실험의 목적은 운영 모델을 바로 고르는 것이 아니라, 군집별로 어떤 피처 조합이 잘 먹히는지 찾는 것이었습니다.  
여기서 `cluster01`은 출퇴근과 교통 접근성 중심 조합이 가장 좋았고, `cluster02`는 야간·주말·공휴일전날과 주거형 특성을 담은 경량 조합이 가장 좋았습니다.  
반면 대표 15개 전체에 공통으로 `weather_full`을 넣은 비교안은 점수 개선이 거의 없었습니다.  
즉 대표 15개 실험의 핵심 성과는 ‘최종 운영 모델 확정’이 아니라 ‘군집별로 유효한 피처 축 발굴’에 있었습니다.”

### 유의점

- `대표 15개 실험 = 탐색용`이라는 점을 다시 고정한다.
- `cluster01`, `cluster02`만 짧고 분명하게 설명한다.

## 챕터 8. 강남구 전체 161개 스테이션 실험

### 실제 스피치

“실제 운영 모델 선택은 강남구 전체 161개 공통 스테이션 기준으로 진행했습니다.  
원본 기준선 모델에서 시작해 정적 피처를 복원한 단일 모델, 군집별 정적 피처를 강조한 모델, 날씨-시간대 상호작용 피처를 붙인 모델, 그리고 군집 분기 적용 모델까지 비교했습니다.  
결과적으로 가장 안정적이었던 조합은 `static enriched + weather_full interaction`이었습니다.  
즉 정적 입지 피처를 붙인 단일 모델에, 비와 강수량을 출퇴근·점심·야간 시간대와 결합한 상호작용 피처를 더한 조합이 최종 운영안으로 선택됐습니다.”

### 유의점

- 이 챕터에서는 `161개 기준에서 고른다`는 점을 가장 먼저 말한다.
- `routing`이 왜 탈락했는지까지 같이 설명할 준비를 한다.

## 챕터 9. 오류 분석

### 실제 스피치

“오류 분석에서는 상위 오류 스테이션이 특정 시간대에 집중된다는 점을 확인했습니다.  
특히 저녁 피크와 아침 피크, 즉 출근·귀가 전환 시간대에서 과대 혹은 과소 예측이 크게 나타났습니다.  
이 결과는 단순 날씨 피처보다 날씨와 시간대를 결합한 상호작용 피처가 더 의미 있다는 근거가 됐습니다.  
또 이런 오류 패턴은 나중에 부족 위험이나 재배치 우선순위 같은 후처리 규칙을 설계할 때도 중요한 기준이 됩니다.”

### 유의점

- `왜 weather interaction이 유효했는가`를 오류 분석과 연결해서 설명한다.
- 차트는 `어느 시간대에서 흔들렸는지`를 보여주는 근거라는 점을 짚는다.

## 챕터 10. 최종 모델 및 성능 요약

### 실제 스피치

“최종 결론은 명확합니다.  
강남구 전체 운영 기준선은 `LightGBM` 기반의 단일 모델이고, 조합은 `static enriched + weather_full interaction`입니다.  
대표 15개 군집 실험은 최종 운영 모델 자체보다 피처 발굴과 해석 근거에 더 큰 의미가 있었고, 161개 전체 기준에서는 군집 분기 적용보다 단일 모델이 더 안정적이었습니다.  
즉 군집화는 피처 발굴과 설명에 유효했고, 실제 운영안은 단일 모델이 더 강했다는 것이 이번 분석의 핵심 결론입니다.”

### 유의점

- `군집 실험은 실패`라고 말하지 말고, `역할이 달랐다`고 정리한다.
- 운영 모델과 해석 모델의 역할 분리를 다시 한 번 강조한다.

## 챕터 11. 서비스 적용 방안

### 실제 스피치

“이 장은 아직 준비 중이지만, 현재까지 확정된 방향은 분명합니다.  
예측 결과는 단순 대여량 숫자로 끝나는 것이 아니라, 예상 잔여 자전거 수, 부족 위험, 재배치 우선순위 같은 서비스용 지표로 후처리되어야 합니다.  
즉 모델이 바로 서비스 화면으로 가는 것이 아니라, 예측값을 운영 판단용 값으로 한 번 더 바꾸는 단계가 필요합니다.”

### 유의점

- 아직 준비 중인 장이므로 방향만 짧게 말한다.
- 너무 세부적인 API 설명으로 들어가지 않는다.

## 챕터 12. 결론 및 향후 과제

### 실제 스피치

“이번 분석을 통해 강남구 따릉이 스테이션 수요 예측에서 군집화, 대표 15개 탐색, 161개 운영 기준선 선택이라는 흐름을 하나로 정리할 수 있었습니다.  
앞으로의 과제는 현재 확정한 운영 모델을 서비스 출력 구조와 연결하고, 실제 앱과 관리 화면에서 해석 가능한 지표로 바꾸는 것입니다.  
또 향후에는 더 정밀한 후처리 규칙과 운영 시나리오를 붙여, 예측값이 실제 재배치 판단에 바로 쓰일 수 있게 만드는 것이 다음 단계입니다.”

### 유의점

- 결론은 `무엇을 확정했는가`와 `다음 단계가 무엇인가` 두 줄기로 닫는다.
- 이 장에서는 점수보다 전체 흐름 요약이 더 중요하다.

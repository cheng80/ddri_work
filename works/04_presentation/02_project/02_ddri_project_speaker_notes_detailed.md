# 강남구 따릉이 수요 예측 프로젝트 중간 발표 스피치 노트

작성일: 2026-03-12  
대상 문서: `works/04_presentation/02_project/01_ddri_project_temp_presentation.md`  
용도: 전체 프로젝트 발표 자료를 페이지별로 바로 읽을 수 있는 실제 스피치와, 따로 확인할 유의점을 분리해 정리한 노트

## 페이지 1. 표지 및 발표 범위

### 실제 스피치

“이번 발표에서는 강남구 따릉이 Station별 하루 대여량 예측 프로젝트의 현재 진행 상황을 정리해서 공유드리겠습니다.  
오늘 발표는 최종 모델 결과 발표라기보다, 문제 정의와 데이터 기준을 확정하고, 군집화와 예측용 데이터셋 구축까지 어디까지 왔는지를 중간 점검하는 자리입니다.”

### 유의점

- 첫 페이지에서는 결과보다 발표 범위를 먼저 고정한다.
- “중간 점검”이라는 표현으로 현재 단계 성격을 분명히 한다.

## 페이지 2. 프로젝트 목적

### 실제 스피치

“프로젝트 목표는 강남구 Station별 하루 따릉이 대여량을 예측하는 것입니다.  
시간, 날씨, 공휴일, 주변 환경, 과거 수요를 함께 반영하고, 예측 결과는 이후 웹 지도 기반 서비스로 연결하는 것을 목표로 하고 있습니다.  
또 적정 보유 대수는 1차 예측 target이 아니라, 대여량 예측 이후 운영 해석 지표로 분리해서 접근하고 있습니다.”

### 유의점

- `대여량 예측`과 `적정 보유 대수 해석`을 혼동하지 않게 분리해서 말한다.
- 목표를 길게 확장하지 않고 현재 범위 안에서 정리한다.

## 페이지 3. 문제 정의

### 실제 스피치

“현재 1차 문제는 회귀 문제로 정의했고, target은 `rental_count`, grain은 `station_id + date`로 확정했습니다.  
일 단위 예측을 먼저 선택한 이유는 현재 내부 기획 문서 흐름과 웹 조회 방식이 날짜 기준과 가장 잘 맞고, 데이터 구조도 시간 단위보다 일 단위가 더 안정적이기 때문입니다.”

### 유의점

- `회귀`, `target`, `grain` 3개 키워드를 명확히 말한다.
- 일 단위 선택 이유는 “웹 흐름 + 데이터 안정성” 두 줄로 요약한다.

## 페이지 4. 데이터 구성

### 실제 스피치

“현재 사용 중인 내부 데이터는 강남구 따릉이 이용 이력과 대여소 마스터, 그리고 여기서 만든 station-day 집계 테이블입니다.  
외부 데이터로는 날씨, 공휴일, 공원·지하철·버스정류장 위치 정보까지 확보했습니다.  
즉 지금 단계에서는 모델 학습에 들어갈 수 있을 정도로 핵심 입력 구조는 이미 준비된 상태입니다.”

### 유의점

- 내부 데이터와 외부 데이터를 나눠서 말한다.
- 아직 쓰지 않는 보관 데이터까지 언급하지 않는다.

## 페이지 5. 데이터 전처리

### 실제 스피치

“전처리에서는 결측치 제거, 0 이하 시간과 거리 제거, 공통 기준 밖 대여소 제거, 강남구 기준 밖 반납 제거를 적용했습니다.  
중복 로그와 극단치 후보도 점검했지만, 극단치는 1차 군집화에서 임의로 제거하지 않았습니다.  
특히 같은 대여소로 반납된 이동은 이상치가 아니라 실제 이용이 발생한 로그이므로, 제거하지 않고 운영 해석용 지표 후보로 유지했습니다.”

### 유의점

- 왼쪽 차트는 전처리 전후 규모 변화, 오른쪽 차트는 제거 사유 설명용이다.
- self-return은 “이상치 아님”으로 짧게 정리한다.
- 전처리 페이지에서 모델 성능 이야기로 넘어가지 않는다.

## 페이지 6. 탐색적 데이터 분석

### 실제 스피치

“EDA 단계에서는 대여소별 평균 대여량과 변동성을 먼저 확인했습니다.  
그 결과 강남구 대여소는 수요 규모 차이가 뚜렷했고, 일부 대여소는 평균 대여량과 변동성이 모두 높은 특성을 보였습니다.  
즉 단순 평균 비교만으로도 유형화가 필요하다는 점이 확인됐고, 이것이 군집화와 feature 설계의 출발점이 됐습니다.”

### 유의점

- 이 페이지는 결과 나열보다 “왜 군집화가 필요했는가”를 연결하는 역할이다.
- 세부 수치보다 “수요 규모 차이” 메시지를 강조한다.

## 페이지 7. Feature Engineering 및 도입 근거

### 실제 스피치

“현재 feature는 시간, 날씨, 공휴일, 과거 수요, 정적 환경, 군집 label까지 묶어서 구성하고 있습니다.  
또 운영 해석을 위해 `return_count`, `same_station_return_count`, `same_station_return_ratio`, `net_flow`를 별도 지표로 만들었습니다.  
여기서 첫 번째 차트는 self-return과 순유출입 같은 운영 보조 지표가 실제로 어느 정도 수준인지 보여줍니다.  
이 차트에서는 학습 구간과 테스트 구간 모두 self-return이 아주 드문 예외가 아니라는 점, 그리고 순유출입이 대여량과 별도로 해석할 가치가 있다는 점을 확인할 수 있습니다.  
두 번째 히트맵은 대여량, 반납량, self-return 비율, 순유출입, 날씨 변수 사이의 상관 구조를 보여줍니다.  
즉 어떤 변수는 서로 가깝고, 어떤 변수는 별도로 봐야 하는지를 설명하는 근거로 활용할 수 있습니다.”

### 유의점

- 첫 번째 차트는 `운영 보조 지표의 필요성`, 두 번째 차트는 `입력 구조의 관계` 설명용이다.
- 히트맵에서는 상관이 높은 변수와 별도 의미를 갖는 변수를 구분해서 말한다.
- POI 보류 항목은 마지막에 짧게만 언급한다.

## 페이지 8. 머신러닝 모델 및 실험 설계

### 실제 스피치

“현재는 baseline target과 grain이 확정됐고, 학습/테스트용 베이스라인 데이터셋까지 생성이 끝난 상태입니다.  
또 기존 운영 스테이션과 신규 스테이션을 분리해서 평가하겠다는 원칙도 정리했습니다.  
다음 단계에서는 선형 회귀, Random Forest, XGBoost, LightGBM 순으로 baseline과 성능 비교를 진행할 계획입니다.”

### 유의점

- 여기서는 “모델 학습 직전 상태”라는 점을 강조한다.
- 아직 없는 성능 결과를 예고하듯 말하지 않는다.

## 페이지 9. 평가 지표

### 실제 스피치

“평가 지표는 현재 `NMAE`를 기준으로 잡고 있습니다.  
그리고 메인 평가셋은 기존 운영 스테이션 중심으로 두고, 신규 스테이션은 cold-start 예외 집합으로 분리해서 관리할 예정입니다.  
이렇게 해야 같은 조건에서 모델별 성능을 비교할 수 있습니다.”

### 유의점

- 지표 이름보다 평가셋 분리 원칙을 함께 말해야 맥락이 잡힌다.
- 신규 스테이션은 메인 평가셋에 섞지 않는다는 점을 분명히 한다.

## 페이지 10. 시스템 구조

### 실제 스피치

“캔버스 기준으로 보면 시스템은 데이터 수집, 전처리, EDA, Feature Engineering, 모델 학습, 예측 생성, API, 웹 시각화 순서로 이어집니다.  
현재까지는 데이터 수집, 전처리, 1차 EDA, 군집화와 환경 해석, 예측용 베이스라인 데이터셋 구축까지 완료된 상태입니다.  
즉 다음 단계는 모델 학습과 API, 웹 연결 쪽에 더 집중하면 됩니다.”

### 유의점

- 완료 범위와 이후 범위를 명확히 나눠 말한다.
- 시스템 구조 페이지에서 세부 구현 기술로 깊게 들어가지 않는다.

## 페이지 11. 웹 시스템 구성

### 실제 스피치

“웹에서는 Station 선택, 날짜 입력, 예측 대여량 조회, 지도 기반 시각화를 목표 기능으로 두고 있습니다.  
지금은 Folium 기반 군집 지도와 예측용 데이터 구조가 준비돼 있기 때문에, 이후 `/predict` 형태 API와 Flutter 지도 연결로 확장하는 방향이 자연스럽습니다.”

### 유의점

- 현재 구현 완료처럼 들리지 않게 “향후 구현 방향” 톤을 유지한다.
- 준비된 선행 결과와 앞으로 만들 기능을 구분한다.

## 페이지 12. 프로젝트 일정

### 실제 스피치

“일정상으로는 데이터 구조 확인과 수집은 완료했고, 현재는 EDA와 Feature Engineering을 진행 중입니다.  
바로 다음 단계는 모델 학습과 중간 발표 준비, 이후에는 최종 모델과 웹 구현, 보고서와 영상 제작 순서로 이어집니다.  
즉 지금은 분석 기준을 고정하고, 모델 학습 직전까지 준비한 상태라고 보시면 됩니다.”

### 유의점

- 일정은 세부 날짜보다 현재 위치를 설명하는 용도로 사용한다.
- “지연”이나 “부족” 프레임보다 현재 준비 상태를 명확히 말한다.

## 페이지 13. 최종 산출물

### 실제 스피치

“최종 제출 기준 산출물은 분석 및 ML 보고서, 시스템 설계 문서, 웹 프로젝트, 시연 영상입니다.  
현재까지는 군집화 노트북과 스크립트, 공휴일과 날씨 수집 노트북, 예측용 베이스라인 데이터셋, 그리고 발표용 차트와 지도 문서까지 확보한 상태입니다.”

### 유의점

- 최종 산출물과 현재 확보한 중간 산출물을 구분해서 말한다.
- 아직 없는 산출물을 이미 준비된 것처럼 말하지 않는다.

## 페이지 14. 기대 결과 및 현재 중간 결론

### 실제 스피치

“현재까지 확인된 중간 결과를 정리하면, 강남구 대여소는 일반수요형과 고수요형으로 구조화됐고, 예측용 station-day 베이스라인 데이터셋까지 구축됐습니다.  
즉 문제 정의, 데이터셋 구조, 전처리 원칙, 군집 label과 환경 feature까지 모두 정리됐고, 이제 이후 단계는 모델 성능 비교와 웹 연결 구현에 집중할 수 있습니다.”

### 유의점

- 결론 페이지에서는 새로운 세부 근거보다 전체 정리 메시지를 준다.
- “모델 성능 비교와 웹 연결”이 다음 중심 단계라는 점으로 마무리한다.

## 페이지 15. 부록. 군집화 핵심 결과

### 실제 스피치

“부록에는 군집화 핵심 결과를 다시 넣었습니다.  
질문이 들어오면 여기서 먼저 왼쪽 PCA 분포를 보여주면서 군집이 실제로 어떻게 나뉘는지 설명하고,  
그다음 오른쪽 환경 비교 차트로 넘어가 고수요형이 교통 접근성이 더 좋은 대여소군이라는 점을 다시 설명하면 됩니다.”

### 유의점

- 부록은 질문 대응용이다.
- PCA와 환경 차트를 순서대로 설명하고, 본문 메시지를 반복 정리하는 용도로 쓴다.

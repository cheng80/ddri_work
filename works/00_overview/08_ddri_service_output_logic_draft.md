# Ddri Service Output Logic Draft

작성일: 2026-03-14  
목적: 수요예측 결과를 운영자용 재배치 판단과 일반 사용자용 잔여 자전거 안내로 연결하는 1차 로직 초안을 정리한다.

## 1. 왜 이 문서가 필요한가

현재 ML의 1차 직접 예측값은 `rental_count`다.

하지만 실제 서비스에서 필요한 출력은 아래와 같이 다르다.

- 운영자: 어떤 스테이션을 언제 재배치해야 하는가
- 일반 사용자: 특정 시각에 자전거가 몇 대 남아 있을까

즉, `수요예측값`을 그대로 보여주는 것만으로는 서비스 요구를 만족하지 못한다.

이 문서는 예측값을 서비스 출력으로 바꾸는 후처리 로직의 초안이다.

## 2. 기본 전제

현재 프로젝트는 아래 순서로 해석한다.

1. 모델이 `수요(대여량)`를 예측한다
2. 예측 수요를 현재 보유/재고 정보와 결합한다
3. 운영자용 재배치 판단값과 사용자용 잔여 자전거 안내값을 만든다

즉, 서비스 출력은 `예측 모델 + 운영 로직`의 결합 결과다.

## 3. 운영자용 출력 로직 초안

### 3.1 목표

운영자는 아래 질문에 답할 수 있어야 한다.

- 어느 스테이션이 언제 부족해질 가능성이 높은가
- 어느 스테이션에 자전거를 추가 배치해야 하는가
- 어느 스테이션은 회수 후보인가

### 3.2 필요한 입력

- `station_id`
- `date`
- `hour`
- `predicted_rental_count`
- 현재 스테이션 보유 대수 또는 실시간 재고
- 스테이션 용량 정보가 있으면 함께 사용
- 군집 정보
- 과거 평균 수요 또는 변동성 지표

### 3.3 1차 지표 후보

- `predicted_demand`: 해당 시간대 예측 대여량
- `current_bike_stock`: 현재 자전거 수
- `stock_gap = current_bike_stock - predicted_demand`
- `risk_score`: 부족 위험도 점수
- `reallocation_priority`: 재배치 우선순위 등급

### 3.4 간단한 해석 규칙 예시

- `stock_gap`이 크게 음수이면 부족 위험 높음
- `stock_gap`이 크게 양수이면 회수 또는 재배치 공급 후보
- 출근/퇴근 핵심 시간대는 가중치를 더 높게 줌
- 업무지구/주거지구 등 군집 특성에 따라 임계값을 다르게 둘 수 있음

### 3.5 운영자 화면 출력 예시

- 부족 위험 스테이션 Top N
- 회수 후보 스테이션 Top N
- 시간대별 재배치 필요 지도
- 군집별 위험도 비교

## 4. 일반 사용자용 출력 로직 초안

### 4.1 목표

일반 사용자는 아래 질문에 답을 원한다.

- 내가 가려는 시간에 이 스테이션에 자전거가 남아 있을까
- 대여 가능성이 높은가 낮은가

### 4.2 필요한 입력

- `station_id`
- `date`
- `hour`
- `predicted_rental_count`
- 현재 스테이션 보유 대수 또는 직전 시점 재고
- 가능하면 직전 시간 반납 예측 또는 최근 평균 반납량

### 4.3 1차 출력 후보

- `predicted_remaining_bikes`
- `bike_availability_flag`
- `availability_level`

### 4.4 간단한 계산 초안

가장 단순한 1차 구현은 아래 식이다.

`predicted_remaining_bikes = current_bike_stock - predicted_rental_count`

그 다음 결과를 해석한다.

- `predicted_remaining_bikes <= 0` 이면 대여 어려움
- `1 ~ 2대` 이면 부족 위험 높음
- `3대 이상` 이면 대여 가능성 있음

이 방식은 단순하지만, 반납 유입을 반영하지 못하는 한계가 있다.

### 4.5 개선 방향

더 현실적인 방식은 아래와 같다.

`predicted_remaining_bikes = current_bike_stock + predicted_return_count - predicted_rental_count`

또는

`current_bike_stock`이 없다면 최근 재고 상태 또는 평균 재고 수준을 보조 입력으로 사용해야 한다.

## 5. 현재 데이터 기준으로 가능한 1차 구현안

현재 바로 가능한 범위는 아래와 같다.

### 운영자용

- `station-day` 또는 `station-hour` 수요 예측
- 군집 정보 포함
- 시간대 부족 위험 스코어 산출
- 재배치 우선순위 규칙 기반 정렬

### 일반 사용자용

- `station-hour` 예측값 사용
- 특정 스테이션, 특정 시각 기준 수요 예측
- 현재 재고 또는 최근 재고 입력이 있으면 `예상 잔여 자전거 수` 계산 가능

즉, 일반 사용자 기능은 `station-hour` 데이터셋이 핵심 기반이다.

## 6. 현재 부족한 입력

완전한 서비스 로직으로 가려면 아래 정보가 추가로 필요하다.

- 실시간 또는 최근 시점 스테이션별 재고 정보
- 스테이션별 거치 가능 대수
- 가능하면 시간대별 반납 예측값 또는 반납량 추정 규칙

현재는 이 값들이 overview 기준으로 아직 정식 설계되지 않았다.

## 6.1 현재 확보한 실시간 API 후보

사용 가능한 실시간 입력 소스 후보는 아래와 같다.

### 서울시 실시간 도시데이터

- 용도:
  - 실시간 도시 상태, 혼잡/이동 맥락을 서비스 확장 입력으로 검토
  - 운영자 화면의 보조 상황 정보로 활용 가능
- 링크:
  - `https://data.seoul.go.kr/dataList/OA-21285/F/1/datasetView.do`

### 서울시 공공자전거 따릉이 실시간 대여정보

- 용도:
  - 스테이션별 현재 자전거 수 또는 실시간 운영 상태 입력 후보
  - 일반 사용자 `예상 잔여 자전거 수` 계산의 핵심 입력 후보
  - 운영자 재배치 우선순위 계산 시 현재 재고 기준값으로 활용 가능
- 링크:
  - `https://data.seoul.go.kr/dataList/OA-15493/A/1/datasetView.do`

### 현재 판단

이 두 API 중 특히 `서울시 공공자전거 따릉이 실시간 대여정보`는 이후 서비스 구현에서 매우 중요하다.

이유:

- 현재 예측 모델은 `미래 수요`를 준다
- 실시간 API는 `현재 재고/운영상태`를 준다
- 두 값을 결합해야 `예상 잔여 자전거 수`와 `재배치 필요성`을 계산할 수 있다

## 7. 1차 권장 구현 순서

1. `station-day` 모델로 기본 수요 예측 성능 확보
2. 전체 공통 스테이션 `station-hour` 모델로 시간대 수요 예측 실험
3. 따릉이 실시간 API에서 `현재 재고` 입력 구조 확인
4. 운영자용 `risk_score`와 `reallocation_priority` 규칙 정의
5. 일반 사용자용 `predicted_remaining_bikes` 계산식 초안 확정
6. 웹/API 출력 스키마 확정

## 9. 현재 결론

현재 프로젝트에서 가장 현실적인 1차 구조는 아래와 같다.

- 모델은 먼저 `대여량`을 예측한다
- 운영자 기능은 이를 `재배치 위험도`로 변환한다
- 일반 사용자 기능은 이를 `예상 잔여 자전거 수` 또는 `대여 가능 여부`로 변환한다

즉, 수요예측 모델과 서비스 출력 로직은 분리해서 설계하는 것이 적절하다.

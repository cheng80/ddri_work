# Ddri Prediction Target Definition

작성일: 2026-03-11  
최종 갱신일: 2026-03-14  
목적: 현재 예측 파트에서 실제로 사용 중인 타깃, 데이터셋 단위, 서비스 해석 방향을 정리한다.

## 0. 용어 정리

- `subset`(축소 피처 조합)
  - 전체 후보 피처 중 일부만 골라 만든 피처 묶음
- `objective`(학습 목표 함수)
  - 모델이 무엇을 더 잘 맞추도록 학습할지 정하는 기준
- `RMSE objective`
  - 일반 회귀형 학습 목표 함수
- `Poisson objective`
  - 수요량, 건수처럼 `0 이상 count 데이터`에 맞춘 학습 목표 함수

## 1. 현재 결론

현재 프로젝트의 서비스 목표는 아래 두 사용자 문제를 동시에 다룬다.

### A. 운영자 문제

- 어떤 스테이션이 언제 부족해질지 예측
- 재배치 우선순위와 운영 판단 지원

### B. 일반 사용자 문제

- 사용자가 특정 스테이션과 특정 시각을 입력했을 때
- 자전거가 몇 대 남아 있을지 예측하여 안내

이때 분석/ML 단계의 직접 예측값은 우선 `대여량`이며, 서비스 출력은 이 예측값을 후처리해 만든다.

현재 예측 파트는 하나의 데이터셋만 있는 것이 아니라, 아래 세 트랙으로 정리되어 있다.

### 트랙 A. station-day 베이스라인

- 예측 target: `Station별 하루 대여량`
- target 컬럼명: `rental_count`
- 데이터셋 grain: `station-day`
- 문제 유형: 회귀
- 결과 위치: `works/03_prediction/02_data/`

### 트랙 B. 대표 대여소 station-hour 실험

- 예측 target: `Station별 시간대 대여량`
- target 컬럼명: `rental_count`
- 데이터셋 grain: `station-hour`
- 문제 유형: 회귀
- 결과 위치: `works/05_prediction_long/`

### 트랙 C. 전체 공통 스테이션 station-hour 실험

- 예측 target: `Station별 시간대 대여량`
- target 컬럼명: `rental_count`
- 데이터셋 grain: `station-hour`
- 문제 유형: 회귀
- 원본 데이터 위치: `3조 공유폴더/군집별 데이터_전체 스테이션/full_data/`
- 실험 관리 위치: `works/06_prediction_long_full/`
- 1차 baseline 및 학습 목표 함수(objective) 비교 결과: `LightGBM_RMSE_Full`, test RMSE `0.8624`

## 2. 왜 이렇게 정리하는가

기존 문서 기준으로는 `station-day`가 1차 목표였다. 이 기준은 여전히 유효하다.

하지만 실제 작업은 여기서 더 진행되었다.

1. 전체 공통 대여소 대상 `station-day` 베이스라인 데이터셋 생성 완료
2. 대표 대여소 15개 대상 `station-hour` long-format 데이터셋 생성 완료
3. 전체 공통 스테이션 161개 대상 `station-hour` long-format 데이터셋 생성 완료

즉, 현재 상태를 최신으로 반영하면 `station-day만 존재`한다고 쓰는 것은 맞지 않는다.

## 3. 현재 실제 산출물

### 3.1 station-day 베이스라인 데이터셋

파일:

- `works/03_prediction/02_data/ddri_station_day_train_baseline_dataset.csv`
- `works/03_prediction/02_data/ddri_station_day_test_baseline_dataset.csv`

특징:

- 공통 대여소 기반
- 일 단위 집계

파일:

- `3조 공유폴더/대표대여소_예측데이터_15개/raw_data/ddri_prediction_long_train_2023_2024.csv`
- `3조 공유폴더/대표대여소_예측데이터_15개/raw_data/ddri_prediction_long_test_2025.csv`

특징:

- 대표 대여소 15개만 사용
- 시간 단위 집계
- `station_group` 포함
- 군집 유형별 패턴 설명과 보조 실험용

### 3.3 전체 공통 스테이션 station-hour 데이터셋

파일:

- `3조 공유폴더/군집별 데이터_전체 스테이션/full_data/ddri_prediction_long_train_2023_2024.csv`
- `3조 공유폴더/군집별 데이터_전체 스테이션/full_data/ddri_prediction_long_test_2025.csv`

특징:

- 학습/테스트 공통 스테이션 `161개`
- 시간 단위 집계
- 운영자 재배치 판단용 시간대 수요 분석 기반
- 일반 사용자 `특정 시각 잔여 자전거 예측` 기능의 기반 입력 테이블 후보
- 대표 대여소 15개 실험과 섞지 않기 위해 실험 노트북/지표는 `works/06_prediction_long_full/`에서 별도 관리
- 1차 baseline 노트북과 지표가 이미 생성되어 있음
- `RMSE objective`(일반 회귀형 학습 목표 함수)가 `Poisson objective`(count 데이터형 학습 목표 함수)보다 우세한 것으로 확인됨

## 4. 현재 운영 정의

세 트랙 모두 예측하려는 값은 동일하게 `대여량`이다.

즉,

- 1차 목표는 수요 예측
- 운영자 기능은 수요 예측을 재배치 판단으로 변환
- 일반 사용자 기능은 수요 예측과 재고/보유 추정 로직을 결합해 `예상 잔여 자전거 수` 또는 `대여 가능 여부`로 변환
- `보유 적정 대수`는 수요 예측 결과를 바탕으로 해석하는 2차 운영 지표

## 5. 현재 기준의 우선순위

### 1순위: station-day 베이스라인 모델

이유:

- 기획 문서와 발표 문서의 기본 서사가 날짜 단위 예측과 가장 잘 맞음
- 전체 공통 대여소를 포괄하기 때문에 서비스/운영 관점 설명이 쉬움
- ML 베이스라인 구축을 가장 빨리 진행할 수 있음

### 2순위: 전체 공통 스테이션 station-hour 모델/실험

이유:

- 운영자 재배치 판단은 시간대 수준 정보가 필요함
- 일반 사용자 `특정 시각 잔여 자전거 예측`도 시간대 단위 입력이 필요함
- 전체 스테이션을 포괄해야 실제 서비스 적용성이 높음

### 3순위: 대표 대여소 station-hour 실험

이유:

- 군집 유형별 시간 패턴 비교에 유리함
- 대표 사례 설명용으로 좋음
- 발표/웹 시연용 샘플 시나리오로 확장하기 좋음
- 군집별 본격 최적화 전의 탐색/설명 단계로 해석 가능함

## 5.1 단계 해석

현재 세 트랙은 역할이 다르다.

- `station-day`: 공통 baseline 및 기본 ML 기준선
- 대표 대여소 `station-hour`: 군집별 패턴 설명과 탐색 실험
- 전체 161개 `station-hour`: 서비스용 공통 baseline

즉, 군집을 나눈 최종 목적을 예측 파트까지 밀어붙이려면 다음 단계는 `군집별 축소 피처 조합(subset) 기반 피처/모델 최적화`여야 한다.

## 8. 현재 연결 기준

현재 예측 파트에서 연결 가능한 주요 군집화 정보는 아래와 같다.

- `cluster` 또는 `cluster_label`
- `mapped_dong_code`
- 생활인구 계열 요약값
- 반납 시간대 기반 비율/순유입 관련 요약값
- 환경 보강 실험 파생 변수 후보

다만 실제 사용 범위는 데이터셋 트랙별로 다르다.

- station-day: 운영 지표와 일 단위 요약 피처 중심
- station-hour: 시간대 수요 패턴 피처 중심

## 9. 일반 사용자 기능 해석

일반 사용자가 원하는 것은 사실상 아래 질문이다.

- “내가 가려는 시간에 이 스테이션에 자전거가 남아 있을까?”

이를 위해서는 단순 수요 예측만으로는 부족할 수 있다.

따라서 후속 단계에서 아래 중 하나를 선택해야 한다.

1. 수요 예측 + 현재 보유 대수/재고 정보를 결합해 `예상 잔여 자전거 수` 계산
2. 직접 `잔여 자전거 수` 또는 `대여 가능 여부`를 예측하는 별도 모델 설계

현재 데이터 구조상 1번이 1차 구현안으로 더 현실적이다.

추가로, 현재 확보한 실시간 입력 후보는 아래와 같다.

- 서울시 실시간 도시데이터
- 서울시 공공자전거 따릉이 실시간 대여정보

특히 `서울시 공공자전거 따릉이 실시간 대여정보`는 `current_bike_stock` 역할을 할 가능성이 높기 때문에, 일반 사용자용 `예상 잔여 자전거 수` 계산에서 핵심 입력 후보로 본다.

## 10. 현재 남은 결정 사항

다음 단계에서 확정해야 할 것:

- station-day 모델의 validation 분리 방식
- 전체 공통 스테이션 station-hour 모델을 실제 학습까지 확장할지 여부
- 대표 대여소 station-hour 데이터를 실제 모델 학습까지 확장할지 여부
- `보유 적정 대수`를 어떤 후처리 지표로 계산할지 여부
- `예상 잔여 자전거 수` 계산에 필요한 현재 재고 입력을 어떤 방식으로 둘지 여부
- 웹에서 운영자 화면과 일반 사용자 화면을 어떻게 분리할지 여부

## 11. 보고용 요약 문장

예시:

“현재 예측 파트는 세 개의 트랙으로 운영된다. 전체 공통 대여소를 대상으로 한 station-day 베이스라인 데이터셋은 기본 ML 모델 학습용으로 사용하고, 전체 공통 스테이션 161개 기준 station-hour 데이터셋은 운영자 재배치 지원과 일반 사용자 특정 시각 잔여 자전거 예측 기능으로 확장하기 위한 핵심 시간대 데이터셋으로 사용한다. 대표 대여소 15개 기준 station-hour 데이터셋은 군집 유형별 패턴 설명과 보조 실험용으로 사용한다. 모든 트랙의 1차 예측 타깃은 `rental_count`이며, 재배치 필요성이나 예상 잔여 자전거 수는 이후 후처리 해석 단계에서 연결한다.”

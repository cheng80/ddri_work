# Ddri Prediction Target Definition

작성일: 2026-03-11
목적: 예측 문제의 target과 데이터셋 grain을 고정하여 이후 ML/웹 설계를 일관되게 진행한다.

## 1. 결론

현재 프로젝트의 1차 예측 문제는 아래와 같이 고정한다.

- 예측 target: `Station별 하루 대여량`
- target 컬럼명(분석용): `rental_count`
- 데이터셋 grain: `station-day`
- 문제 유형: 회귀

즉, 특정 날짜에 특정 대여소에서 발생할 `하루 총 대여 건수`를 예측하는 문제로 정의한다.

## 2. 왜 이 기준으로 고정하는가

문서 전반을 대조하면 `보유 적정 대수` 표현이 일부 등장하지만, 실제 기획 문서와 데이터 수집 문서는 대부분 아래 문제를 기준으로 작성되어 있다.

- Station별 대여량 예측
- 하루 대여량 예측
- `rental_count`를 target으로 정의
- 날짜 입력 후 예측 조회

따라서 현재 단계에서 가장 일관된 문제 정의는 `Station별 하루 대여량 예측`이다.

## 4. `보유 적정 대수`와의 관계

현재 프로젝트 설명에는 `운행에 필요한 적합한 대수`라는 운영 목적 표현이 있다.

이 표현은 완전히 버리는 것이 아니라, 아래처럼 해석하는 것이 적절하다.

- 1차 ML target: `하루 대여량`
- 2차 운영 해석: 예측 대여량을 기반으로 `필요 보유 대수/재배치 필요성`을 설명

즉,

- 예측 모델은 먼저 `수요(대여량)`를 맞춘다.
- 그 다음 운영 관점에서 `적정 보유 대수`나 `재배치 우선순위`를 해석 지표로 연결한다.

이렇게 하면 현재 보유 데이터와 문서 구조에 가장 잘 맞는다.

## 5. 왜 `station-day` grain을 선택하는가

### 장점

- 현재 문서들이 대부분 `하루 대여량` 기준으로 작성되어 있음
- 군집화 결과와 연결하기 쉬움
- 날씨/공휴일/생활 패턴 feature를 결합하기 용이함
- 웹에서 날짜 기반 조회 기능과 잘 맞음
- 초기 베이스라인 모델 구축이 단순함

### `station-hour`를 바로 쓰지 않는 이유

- 시간 단위 예측은 더 세밀하지만 데이터셋 설계와 feature 관리가 복잡해진다.
- 현재 기획 문서와 API 예시는 날짜 단위 조회에 더 가깝다.
- 1차 산출물과 일정상 일 단위가 더 안정적이다.

## 6. 최종 정의

### 예측 입력

- `station_id`
- `date`
- 시간/요일/월 feature
- 날씨 feature
- 공휴일 feature
- 과거 수요 feature
- 환경 feature
- 군집 label(후보)

### 예측 출력

- `predicted_rental_count`

### 예측 단위

- 대여소 1개
- 날짜 1일

## 7. 후속 과제

현재 정의를 기준으로 다음 단계에서 결정해야 할 것:

- `return_count`를 보조 지표로 같이 볼지 여부
- 군집 label을 예측 feature에 포함할지 여부
- `보유 적정 대수`를 후처리 파생 지표로 계산할지 여부
- 차후 2차 확장으로 `station-hour` 예측을 별도 수행할지 여부

## 8. 보고서/PPT에 넣을 수 있는 문장

예시:

“프로젝트의 1차 예측 문제는 Station별 하루 대여량 예측으로 정의하였다. 문서 전반에서 target 변수는 `rental_count`로 일관되게 제시되어 있으며, 웹 서비스 역시 날짜 기반 예측 조회 구조를 전제로 한다. 운영 관점의 ‘적정 보유 대수’는 이 수요 예측 결과를 기반으로 해석하는 2차 지표로 다루는 것이 현재 데이터와 일정에 가장 적합하다.”

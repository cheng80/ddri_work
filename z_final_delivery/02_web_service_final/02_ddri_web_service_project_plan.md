# DDRI 웹서비스 프로젝트 기획서

작성일: 2026-03-16  
목적: 별도 Git 저장소로 분리할 DDRI 웹서비스 프로젝트의 범위, 사용자, 기능, 개발 원칙을 정의한다.

## 1. 프로젝트 개요

DDRI 웹서비스는 강남구 따릉이 대여소 데이터를 바탕으로 다음 두 목적을 지원한다.

- 일반 사용자:
  - 특정 대여소와 시간 기준의 대여 가능성 확인
- 운영 관리자:
  - 재고 부족 가능성이 높은 대여소를 빠르게 파악하고 재배치 판단 지원

이 서비스는 분석/ML 프로젝트의 결과를 직접 노출하는 것이 아니라, 예측 결과와 실시간 재고를 결합한 서비스형 정보를 제공하는 것을 목표로 한다.

## 2. 개발 방식

현재 분석 산출물이 계속 보강 중이므로 개발은 `투트랙`으로 진행한다.

- 트랙 A. 분석/모델 산출물 확정
  - 예측 결과 보강
  - 실시간 API 예외 규칙 고정
  - 서비스용 출력 필드 정리
- 트랙 B. 웹서비스 개발
  - 화면 구조 설계
  - API 계약 정의
  - Flutter 웹 프론트엔드 구현

두 트랙의 접점은 아래 3개로 제한한다.

- API 응답 스키마
- 161개 대상 스테이션 lookup
- 예외 스테이션 처리 규칙

## 3. 대상 사용자

### A. 일반 사용자

목표:

- 지금 또는 특정 시각에 특정 대여소에서 자전거를 빌릴 수 있는지 확인

핵심 행동:

- 대여소 검색
- 날짜/시간 선택
- 예상 잔여 자전거 수 확인
- 대여 가능 여부 확인

### B. 운영 관리자

목표:

- 부족 가능성이 높은 대여소를 선별하고 재배치 우선순위를 확인

핵심 행동:

- 시간 기준 전체 대여소 위험도 확인
- 재고 부족 후보 정렬
- 예외 스테이션 분리 확인
- 군집 또는 조건별 필터링

### C. 통계 조회 사용자

목표:

- 서비스 전체 경향과 군집/시간대별 추이를 확인

핵심 행동:

- 시간대별 수요 추이 확인
- 군집별 비교 확인
- 재고 부족 경향 요약 확인

## 4. 핵심 페이지 범위

현재 필요한 페이지는 아래 3개로 고정한다.

### 1. 사용자 페이지

- 경로: `/user`
- 기능:
  - 대여소 검색
  - 시간 선택
  - 현재 자전거 수 표시
  - 예측 대여량 표시
  - 예상 잔여 자전거 수 표시
  - 대여 가능 여부 배지 표시

### 2. 관리자 페이지

- 경로: `/admin`
- 기능:
  - 전체 대여소 목록 조회
  - 현재 재고 / 예측 수요 / 재고 차이값 표시
  - 위험 점수 및 우선순위 정렬
  - 군집 필터
  - 예외 스테이션 구분 표시

### 3. 통계 페이지

- 경로: `/stats`
- 기능:
  - 시간대별 수요 추이 차트
  - 군집별 비교 차트
  - 대여소별 요약 통계
  - 운영 지표 요약 카드

## 5. 진입 방식과 라우팅 원칙

- 초기 진입점은 링크 기반 페이지로 구성
- Flutter Web에서 `GetRoute` 기반으로 페이지 구분
- 권한 체계는 초기에는 복잡하게 두지 않음
- 1차 버전은 읽기 중심 서비스로 두고, 저장이 필요한 사용자 개인 데이터는 도입하지 않음

권장 기본 경로:

- `/user`
- `/admin`
- `/stats`

## 6. MVP 범위

1차 MVP는 아래 범위까지만 구현한다.

- 사용자 페이지 조회 기능
- 관리자 페이지 목록/정렬/필터 기능
- 통계 페이지 기본 차트 3~5종
- 예외 스테이션 메시지 처리
- 실시간 재고와 예측 결과를 결합한 응답 표시

이번 단계에서 제외하는 항목:

- 회원가입
- 복잡한 권한 체계
- 사용자별 개인화 저장
- 푸시 알림
- 대규모 지도 상호작용
- 모바일 앱 전용 기능

## 7. 기능 우선순위

### 최우선

- 대여소 단건 조회
- 운영자용 재배치 우선순위 표
- 통계 요약 카드

### 차순위

- 군집 필터
- 날짜/시간 선택 고도화
- 차트 상호작용

### 후순위

- 즐겨찾기
- 사용자별 설정 저장
- 알림

## 8. 데이터 원천

- 예측 데이터:
  - 분석/ML 산출물 기반 서버 예측 결과
- 실시간 재고:
  - `OA-15493 bikeList`
- 스테이션 매핑:
  - `tbCycleStationInfo`
- 서비스 대상 lookup:
  - 161개 스테이션 기준 매핑 테이블

## 9. 저장 전략

현재 서비스는 사용자 계정 기반 저장보다 공용 데이터 조회 서비스에 가깝다.

따라서 1차 저장 전략은 아래처럼 단순하게 둔다.

- 서버 저장:
  - 스테이션 마스터
  - 예측 결과 테이블
  - 실시간 재고 캐시
  - 통계 집계 결과
  - 시스템 로그
- 클라이언트 저장:
  - 최근 선택 대여소
  - 최근 조회 시간
  - 필터 상태

클라이언트 저장은 `localStorage` 수준으로 제한한다.

## 6. 전체 중요 feature 해석

`GradientBoosting` 기준 전체 중요도 상위 feature는 아래였다.

1. `hour`
2. `lag_168h`
3. `is_night_hour`
4. `lag_24h`
5. `rolling_mean_24h`
6. `rolling_mean_168h`
7. `precipitation`
8. `lag_1h`
9. `temperature`
10. `is_weekend`

이 결과가 의미하는 건 명확하다.

- 외곽 주거형은 `정적 상권 변수`보다 `시간 패턴 + 반복 패턴 + 야간성`이 훨씬 중요하다
- 정적 지형 변수는 성능을 아주 크게 끌어올리진 못하지만, 3개 대여소를 같이 학습할 때 보조 설명력은 있다
- 따라서 모델링 중심축은 결국 `lag/rolling + 시간대 + 야간/주말 + 날씨`로 잡는 게 맞다

## 7. 대여소별로 우선 봐야 하는 feature

중요한 포인트는 아래다.

- `3개 대여소를 같이 학습하는 군집 모델`에서는 정적 입지 feature가 의미가 있다
- `대여소별 개별 모델`로 쪼개면 정적 입지 feature는 거의 상수가 되므로 의미가 크게 줄어든다

즉 대여소별 추천 feature는 주로 `동적 feature` 중심으로 읽는 게 맞다.

### 7.1 `2359` 국립국악중,고교 정문 맞은편

관찰된 패턴:

- 평균 대여량: `0.7908`
- `lag_168h` 상관: `0.3306`
- `lag_24h` 상관: `0.2997`
- `is_night_hour` 효과: `-0.8299`
- `is_rainy` 효과: `-0.2770`
- `heavy_rain_flag` 효과: `-0.5848`

대여소별 중요 feature 상위:

- `hour`
- `lag_168h`
- `is_night_hour`
- `lag_24h`
- `precipitation`
- `lag_1h`
- `rolling_mean_24h`
- `is_weekend`

- 이 대여소는 `주간 반복성`이 강하다
- 야간 수요 감소가 매우 크다
- 비가 오면 수요가 확실히 줄어든다

권장:

- 이 대여소는 `hour + lag_168h + is_night_hour + precipitation` 조합을 우선 봐야 한다

### 7.2 `2392` 구룡산 입구 (구룡산 서울둘레길 입구)

## 8. 실제 실험 순서 추천

가장 깔끔한 순서는 아래다.

### Step 1. 군집 모델 baseline

`외곽 주거형` 3개 대여소를 같이 묶어서 아래로 시작:

- 기본 15개 원본 컬럼
- protocol 기준 lag/rolling 추가
- 모델: `LinearRegression`, `LightGBM_RMSE`

### Step 2. 군집 모델 round2

여기에 아래 7개만 추가:

- `is_night_hour`
- `is_weekend`
- `station_elevation_m`
- `elevation_diff_nearest_subway_m`
- `distance_naturepark_m`
- `distance_river_boundary_m`
- `bus_stop_count_300m`

이 단계에서:

- `2023` train
- `2024` validation
- 선택 모델을 `2023+2024` 재학습
- `2025` test

으로 보면 된다.

### Step 3. station-level 보조 분석

군집 모델은 유지하되, 대여소별로는 아래처럼 해석을 붙이면 좋다.

- `2359`: 반복성 + 야간 + 강수
- `2392`: sparse 수요 + 단순 시간/lag 구조
- `3643`: 반복성 + 단기 추세 + 야간

즉 `최종 운영 모델은 군집 모델`로 가져가고,  
`대여소별 해석은 local feature recommendation` 형태로 붙이는 게 가장 자연스럽다.

## 9. 최종 추천안

이번 단계에서 가장 무리 없는 결론은 아래다.

1. `외곽 주거형`의 1순위 모델은 기존 문서 기준 `LightGBM_RMSE`

## 11. 최종 운영안 고정

최종 운영안은 아래로 고정한다.

- 모델: `LightGBM_RMSE`
- feature 세트: `baseline + is_weekend(1개 추가)`
- 모델 선택 기준: `validation_2024` RMSE

고정 이유:

- `is_weekend` 단일 추가가 validation/test 모두 baseline보다 개선
  - validation: `0.786281 -> 0.785635` (RMSE `-0.000646`)
  - test: `0.716329 -> 0.714925` (RMSE `-0.001404`)
- 2개 조합도 개선되지만, 개선 폭이 매우 작아 운영 복잡도를 높일 이유가 약함
- `is_weekend`는 해석이 직관적이고 보고서 설득력이 좋음

운영 fallback:

- 재현 환경이 불안정하거나 결과 변동이 커지면 `baseline`으로 즉시 롤백 가능

## 12. 보고서 문구(바로 사용 가능)

### 12.1 결과 요약 문구

외곽 주거형(3개 대여소) 군집에서는 `LightGBM_RMSE`를 기본 모델로 유지하되,  
`is_weekend` 1개 피처를 추가한 경량 커스텀 구성이 가장 실용적이었다.  
Validation(2024)과 Test(2025)에서 모두 baseline 대비 RMSE가 소폭 개선되어,  
운영 모델은 `baseline + is_weekend`로 확정하였다.

### 12.2 수치 문구

- Baseline: validation RMSE `0.786281`, test RMSE `0.716329`
- Final(`+ is_weekend`): validation RMSE `0.785635`, test RMSE `0.714925`
- 개선폭: validation `-0.000646`, test `-0.001404`

### 12.3 해석 문구

외곽 주거형 군집은 복잡한 다중 정적 피처 조합보다,  
주말 여부처럼 의미가 명확한 단일 달력 피처를 추가하는 방식이  
성능-해석성-운영복잡도의 균형에서 가장 적절했다.  
개선 폭은 크지 않지만 방향성이 일관되어, 본 단계에서는  
경량 커스텀 전략을 채택하는 것이 합리적이다.
